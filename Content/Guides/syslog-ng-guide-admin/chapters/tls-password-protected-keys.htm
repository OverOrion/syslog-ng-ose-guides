Password-protected keys

Starting with &abbrev; version 3.147.0.7, you can use password-protected private keys in the network() and syslog() source and destination drivers.

Restrictions and limitations:

  • Hazard of data loss! If you use password-protected keys, you must provide the passphrase of the password-protected keys every time &abbrev; is restarted (&abbrev; keeps the passphrases over reloads). The sources and destinations that use these keys will not work until you provide the passwords. Other parts of the &abbrev; configuration will be unaffected.

    This means that if you use a password-protected key in a destination, and you use this destination in a log path that has multiple destinations, neither destinations will receive log messages until you provide the password. In such cases, always use disk-based buffering to avoid data loss.

  • The path and the filename of the private key cannot contain whitespaces.

  • Depending on your platform, the number of passwords &abbrev; can use at the same time might be limited (for example, on Ubuntu 16.04 you can store 16 passwords if you are running &abbrev; as a non-root user). If you use lots of password-protected private keys in your &abbrev; configuration, increase this limit using the following command: sudo ulimit -l unlimited
